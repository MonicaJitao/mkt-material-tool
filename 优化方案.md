# 后续优化方案

本文档记录不进入 MVP、但后续值得优化的方向。MVP 优先完成 `重构方案.md` 中定义的核心闭环：Brief 填写、方案确认、批量底图生成、用户选择、HTML 生成、预览编辑、分类保存。

## 优化原则

后续优化应遵循三个原则：

1. 先验证工作流，再扩展功能。
2. 先沉淀模板和数据，再做复杂自动化。
3. 先保证普通用户低门槛使用，再支持高级用户深度控制。

## 近期优化方向

这些方向可以在 MVP 跑通后优先考虑。

### 活动模板管理

MVP 可以先内置少量模板。后续可以增加模板管理能力：

- 新建模板。
- 复制模板。
- 修改模板字段。
- 保存某次成功活动为模板。
- 为模板设置默认风格、尺寸、文案结构和 prompt 组装规则。
- 模板版本管理。

目标是把一次成功的生成经验沉淀成可复用资产。

### Prompt 模板迭代工具

Prompt 是这个项目的核心资产之一。后续可以提供专门的模板调优能力：

- 查看每次生成使用的 prompt。
- 对比不同 prompt 版本的生成效果。
- 标记“好结果”和“坏结果”。
- 将好结果对应的 prompt 保存为模板。
- 为不同海报类型维护不同 prompt 模板。

这部分不需要一开始做复杂界面，但后续非常值得做。

### 更强的 Review Agent

MVP 的 Review Agent 只做基础检查。后续可以增强为更完整的质量评审：

- 检查文字可读性。
- 检查视觉风格是否符合 Brief。
- 检查底图是否有明显文字、乱码或错误地标。
- 检查 HTML 是否有布局溢出。
- 检查品牌色、字体、字号是否符合模板规范。
- 给出修改建议，而不是只返回通过/失败。

这能减少用户反复试错的成本。

### 更细的重新生成控制

MVP 至少支持重生成方案、底图候选和 HTML。后续可以做得更细：

- 只重写文案。
- 只调整版式。
- 只换颜色。
- 只调整底图 prompt 中的风格或构图。
- 选择某张底图后生成多个 HTML 版式候选。

这样可以降低用户“全部重来”的挫败感。

## 中期优化方向

这些方向适合在已经有真实使用反馈后再做。

### 素材库高级管理

素材库后续不应只是文件列表，可以逐步升级为营销资产库：

- 按项目、活动、城市、客户经理、尺寸、状态筛选。
- 支持收藏、归档、废弃。
- 支持查看某张底图关联的 HTML。
- 支持查看某个 HTML 对应的生成方案和 prompt。
- 支持批量改名和批量移动分类。

这对多城市、多客户经理、多活动场景会很有价值。

### 批量生产能力

五一案例里存在明显的批量生产需求。后续可以支持：

- 上传城市/客户经理表格。
- 按城市生成不同底图。
- 同一城市为多个客户经理批量生成不同落款。
- 复用同一底图批量生成多个 HTML。
- 批量检查缺失项。

MVP 不必做完整批量生产，但数据模型要为它留空间。

### 半可视化 HTML 编辑

MVP 做源码编辑即可。后续可以在不引入完整页面搭建器的前提下，增加轻量表单式编辑：

- 修改标题。
- 修改副标题。
- 修改落款。
- 修改主题色。
- 修改字体大小。
- 调整文字区域位置。
- 切换几个预设版式。

这比 GrapesJS 级编辑器轻很多，但对普通用户更友好。

### 生成质量反馈闭环

后续可以让用户对生成结果做简单反馈：

- 喜欢/不喜欢。
- 底图好但版式不好。
- 文案好但视觉不好。
- 地标不准确。
- 文字不清晰。

这些反馈可以用于优化模板、prompt 和默认参数。

## 长期优化方向

这些方向适合在产品形态稳定、使用人数增加后再考虑。

### 多用户与权限

如果后期部署到服务器，需要考虑：

- 用户登录。
- 项目权限。
- 团队共享素材库。
- 管理员配置模型 API Key。
- 不同用户的生成记录隔离。

本地单机 MVP 不需要这部分。

### 成本与额度管理

模型调用会产生费用。后续可以增加：

- 每次生成的模型、token、图片数量记录。
- 项目维度成本统计。
- 用户维度额度限制。
- 批量生成前的成本提醒。
- 失败任务是否计费的记录。

MVP 只需要保留模型和参数记录，不需要做成本面板。

### 服务器部署能力

后期如需部署到服务器，需要补充：

- 对象存储或共享文件存储。
- 后台任务队列。
- 任务并发限制。
- 用户隔离。
- 日志与监控。
- API Key 安全管理。

本地 Web 和服务器 Web 可以共用大部分业务代码，但部署层需要单独设计。

### exe 打包体验优化

如果面向本地用户交付 exe，可以优化：

- 首次启动配置向导。
- 自动检测端口。
- 自动打开浏览器。
- 后台托盘图标。
- 本地配置文件管理。
- 错误日志查看。

这部分应在核心工作流稳定后再做。

## 暂不建议优先投入

以下方向短期内不建议投入太多：

- 完整拖拽式页面搭建器。
- 复杂 DOM 树编辑。
- 多类型文件查看器。
- 内置终端。
- OCR。
- 背景移除。
- 复杂动画编辑。
- 从一开始就做大型团队协作平台。

这些能力容易把项目重新拉回 Capybara 的复杂度，不符合当前“低门槛生成营销海报”的核心目标。

## 推荐迭代顺序

建议后续按以下顺序优化：

1. MVP 核心闭环稳定。
2. 活动模板管理。
3. Prompt 模板迭代工具。
4. Review Agent 增强。
5. 批量生产能力。
6. 素材库高级管理。
7. 半可视化 HTML 编辑。
8. 多用户、成本、部署、exe 体验。

每一阶段都应以真实使用反馈为依据，不要只因为技术上能做就加入功能。
